# publish-changes가 비ASCII(한글) 파일명에서 항상 실패

- Issue ID: `issue-20260724T123122Z-b417c9dc`
- Recorded at: 2026-07-24T12:31:22.096839+00:00
- Reported by: claude-code
- Reported from: agent
- Area: cli
- Severity: high
- Release observed: v1-6ca03e60176f
- Status: open

## Summary

core/publisher.py의 publish_changes가 'git diff --cached --name-only' 출력을 그대로 startswith 비교하는데, git의 core.quotepath 기본 동작으로 비ASCII 파일명이 큰따옴표+8진수 이스케이프로 감싸져 반환되어 _is_allowed_path의 'knowledge/'로 시작 여부 검사가 항상 실패한다. 한글 파일명을 쓰는 이 설치에서는 knowledge/ 경로 변경이 있을 때마다 publish-changes가 100% 'publication staging escaped the configured path boundary'로 실패함.

## Expected result

knowledge/ 아래 파일만 변경했을 때 publish-changes --message가 정상적으로 validate 후 commit해야 한다.

## Actual result

python3 .circled-wiki/bin/circled-wiki.py publish-changes --message '...' 실행 시 knowledge/ 내부 한글 파일명 변경만 있어도 'ERROR: publication staging escaped the configured path boundary'로 실패.

## Impact

한글(비ASCII) 파일명을 쓰는 모든 설치에서 공식 발행 CLI가 사실상 사용 불가하며, Agent가 매번 우회 경로(직접 git add/commit)를 써야 함.

## Reproduction or context

한글 파일명의 knowledge/bundles/**/*.md 를 아무거나 수정한 뒤 publish-changes --message test 실행. git diff --cached --name-only 출력이 큰따옴표로 감싸진 이스케이프 문자열임을 확인 가능(예: "knowledge/bundles/...).

## Related paths or artifacts

- `.circled-wiki/runtime/circled_wiki/core/publisher.py`

## Improvement hint

git 호출에 -z(NUL 구분)를 사용해 quotepath 이스케이프를 우회하거나 core.quotepath=false를 명시적으로 지정하도록 _git 헬퍼를 수정.

## Cause hypothesis

구현 시 ASCII 파일명 기준으로만 테스트되어 git의 quotepath 이스케이프를 고려하지 않은 것으로 추정.
